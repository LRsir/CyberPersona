## 官方确认的控制方式

根据 Xiaomi MiMo 官方文档，`mimo-v2.5-tts` 的 TTS 调用要点是：

1. **合成正文必须放在 `role: assistant` 的 `content`**
2. **风格控制可放在 `role: user` 的 `content`**，用于自然语言控制
3. **音频标签控制写在 `role: assistant` 的正文里**
4. **内置音色通过 `audio.voice` 指定**，如 `茉莉`
5. 请求走 **`/v1/chat/completions`**，不是 `/v1/audio/speech`

所以现在需要区分两层：

### 1. 官方可支持的能力边界

- `role:user`：可放自然语言风格提示
- `role:assistant`：放实际要说的话 + 标签
- `audio.voice`：固定 `茉莉`

### 2. 当前项目的正式链路选择

虽然官方支持两层控制，但**本项目正式链路不采用双重控制**。

正式链路固定为：

- `role:assistant`：实际要说的话 + 少量开头标签
- `naturalStylePrompt`：默认留空
- `audio.voice`：固定 `茉莉`

原因很简单：A/B 实测 + 主观试听都表明，双重控制更容易过演。

但要注意：**遵循官方支持 ≠ 正式链路必须把所有能力都打开**。

## 实验链路：双重控制用法（仅供测试，不用于正式链路） 

### 1. 自然语言风格（user 消息）
控制整体情绪走向，放在 `role: user` 的 `content` 中。

建议：
- 只写 **1-2 句短描述**
- 只描述 **语气 / 节奏 / 情绪 / 强弱**
- 不要写成长篇导演说明

示例：

```
温柔、舒缓，像睡前讲故事一样，语速偏慢，带着淡淡的倦意和安心。
```

### 2. 音频标签（assistant 消息）
控制细粒度切换，放在 `role: assistant` 的 `content` 中：

```
(开心)你知道吗，我今天终于拿到offer了！
(突然低落)可是……我妈说太远了。
(释然，带着笑意)算了，至少我证明了自己可以。
```

### 3. 实验性叠加
user 描述整体 + assistant 标签细调 = 可用于测试模型反应

但这里的“细调”要克制：
- 开头 1 组整体风格标签通常就够
- 中途只在明显转折点插 0-2 个音频标签
- 不要一整句一个标签，不要把标签当脚本写

## 常用标签速查

## 实验模板

### 睡前故事
```
user: 温柔、舒缓，像睡前讲故事一样，语速偏慢，带着淡淡的倦意。
assistant: (温柔)今晚的风很轻……(轻声)小狐狸蜷在老橡树的树洞里……
```

### 好消息分享
```
user: 轻快、兴奋，带着压抑不住的激动，语速偏快，声音明亮上扬。
assistant: (开心)你知道吗！我今天终于……(兴奋)就是那种做梦都会笑醒的感觉！
```

### 安慰/释然
```
user: 温柔但坚定，带着理解和鼓励，语速适中，尾音微微上扬给人力量。
assistant: (温柔)没关系，(释然)至少你努力过了。(带着笑意)下次会更好的。
```

## 注意事项

1. **OpenClaw 原生 TTS 工具不支持标签**
   - `/tts latest` 会把标签当文本读出来
   - 需要用脚本直接调 API 才能保留标签

2. **文本长度**
   - 30-50 字最佳，给情绪展开空间
   - 太短标签展不开，太长控制精度下降

3. **标签密度**
   - 每段情绪变化处再加标签
   - 默认从“少标签”开始
   - 不要整段都是标签，会乱

4. **别滥用导演模式**
   - 官方确实支持更复杂的 director mode
   - 但那属于实验能力边界，不属于当前正式链路
   - 在当前赛博女友场景里，这类长提示很容易把声音推得过满、过演

5. **voice 固定为茉莉**
   - 中文女声，稳定，适合大多数场景
   - 不要换其他预置音色，效果不可预期

6. **标签写法要轻**
   - 只保留开头 1 组整体标签
   - 默认最多 2 个标签
   - 优先像：`（轻声）`、`（温柔）`、`（委屈，嘴硬）`
   - 避免：`（轻声，温柔，慢一点，像在床边哄人睡觉）` 这种过满写法

7. **少用正文内音频动作**
   - 如 `[轻哼]`、`[沉默片刻]`、`[深呼吸]`
   - 在正式链路默认去掉，除非以后单独证明确实提升听感

## 2026-04-29 A/B 实测结论

基于 `mimo-v2.5-tts` + `茉莉` 的 12 组实测：

- `plain` → 平均 3.74s
- `tag_only` → 平均 4.33s
- `style_plus_tag` → 平均 5.44s
- `style_only` → 平均 5.81s

结论：

1. **自然语言 style prompt 的影响最强**，会明显拉长节奏。
2. **tag_only 更克制**，更适合日常陪伴型聊天。
3. **双控制不是默认最优**，只适合少数需要更强安抚感的场景。

结合主观试听反馈，当前进一步确认：**双重控制听感不太好，正式链路不采用。**

因此正式链路采用：

- 默认优先 `tag_only`
- `naturalStylePrompt` 在正式链路默认留空
- 禁止长篇导演模式进入正式语音链路

## 生成层补充规则

不仅请求构造阶段要克制，**上游生成阶段也要一起收紧**：

1. 普通轮次的 `naturalStylePrompt` 默认应为空字符串
2. 正式链路中不要主动生成用于 TTS 的非空 `naturalStylePrompt`
3. 即使实验链路需要非空，也只写 1 句短提示，最多 2 句
4. 不能写成“前半段怎样、后半段怎样”的分镜式说明
5. 如果 `taggedTtsText` 已经有明确整体标签，普通轮次不要再重复补 style
